Final Modeling Project

The final project asks each team to build, test, and defend a mathematical imaging reconstruction method.

Students work in teams of two. Each team chooses one imaging inverse problem, implements at least two reconstruction approaches, compares them experimentally, and explains when each approach should and should not be trusted.

The goal is not only to produce a nice reconstructed image. The goal is to show ownership of the model, the code, the experiments, the failures, and the interpretation.

Central Question

Each project should answer the following question:

Given degraded image data y, a forward model A, and noise or uncertainty, how can we reconstruct an image x, and how do we know when the reconstruction is reliable?

A typical project has the form:

y = A x + \eta,

or a related model adapted to the chosen degradation.

The team must define what x, y, A, and \eta mean in its project. If the forward process is not linear, the team must say so explicitly and explain what replaces the linear operator notation.

The course identity is:

\text{data} + \text{model} + \text{prior} + \text{optimization} \rightarrow \text{reconstruction with limits}.

For learned methods, the training data and architecture become part of the prior. For LLM-assisted work, the oral defense and the reproducibility checks are what establish student ownership.

Expected Timeline

Exact calendar dates will be announced in class. The project is assigned in Week 6 and defended during the final project period.

The syllabus defines the assessed project moments as the proposal in Week 6, the report in Week 14, and the oral defense in Week 15. The intermediate rows below are for feedback and preparation; they do not create additional graded items outside the syllabus.

Stage Expected Progress What Students Should Have
Week 6 Project proposal Team, tentative topic, data source, baseline method, second method, first statement of x, y, forward model, degradation, and reliability question.
Week 7 Planning feedback Refined model statement, data access, and first implementation choices.
Week 8 Baseline preparation A runnable notebook that creates the degraded data and runs a simple baseline.
Week 10 Method development Two reconstruction methods implemented on the same test cases.
Week 12 Evidence development Parameter sweep, quantitative metric, and first robustness or failure test.
Week 14 Project report Complete notebook, written report, claim-evidence table, LLM-use statement, and oral-defense preparation.
Week 15 Presentation and oral defense Final submission plus individual code walkthrough and defense.

Required LLM Use

Use of an LLM is required for this project.

Students may use an LLM for:

  • brainstorming project ideas;
  • understanding mathematical definitions;
  • generating or debugging code;
  • improving plots;
  • organizing the report;
  • checking explanations;
  • preparing for the oral defense.

However, students remain fully responsible for the correctness, reproducibility, and interpretation of the submitted work. An LLM can help produce an artifact, but it cannot replace ownership of the artifact.

Each submission must include an LLM-use statement describing:

  1. which LLM tools were used;
  2. what they were used for;
  3. which generated code or text was accepted, modified, or rejected;
  4. what the team verified independently;
  5. at least one mistake, limitation, or misleading suggestion produced by the LLM.

The LLM-use statement is required but not graded separately. A missing or dishonest LLM-use statement makes the project incomplete.

Core Requirements

Each team must include:

Requirement Description
Forward model Define the image, degradation, measurements, and noise model.
Two reconstruction approaches Compare at least two methods; at least one must come from the course.
Parameter study Sweep at least one important parameter and explain the tradeoff.
Quantitative comparison Use RMSE, PSNR, residual norm, feature contrast, SSIM, or another justified metric.
Visual comparison Show reconstructed images and interpret visible artifacts or improvements.
Robustness test Test noise level, model mismatch, domain shift, missing data, or another stress case.
Failure case Show one case where a method fails, and explain why.
Reproducible notebook Submit a runnable Python notebook with clear inputs, outputs, and plots.
Written report Explain the model, methods, experiments, results, and limitations.
Oral defense Each student must individually defend the whole project.

Submission Package

The final submission should contain four parts.

Part Content
Notebook A runnable notebook that generates or loads the data, runs the methods, produces the figures, and prints the key metrics.
Report A concise written explanation of the model, methods, evidence, limitations, and conclusions.
Claim-evidence table Three to five claims, each paired with evidence and a limitation or failure mode.
LLM-use statement The required disclosure described above.

The notebook and report should agree. Every important figure or numerical result in the report should be reproducible from the notebook.

Notebook Expectations

The notebook should be readable by another student in the class.

It should include:

  • a short problem statement;
  • imports and reproducible random seeds;
  • data loading or data generation;
  • a baseline method;
  • a second reconstruction method;
  • parameter sweeps;
  • quantitative metrics;
  • visual comparisons;
  • at least one robustness test;
  • at least one failure case;
  • comments explaining nontrivial code;
  • a final cell summarizing the main claim and limitation.

A notebook that only shows final images is not enough. The notebook must show how the results were obtained.

Report Structure

The report should be concise and structured.

  1. Problem statement. What imaging problem is being solved?
  2. Forward model. Define x, y, A, and the noise or degradation.
  3. Methods. Describe the reconstruction algorithms.
  4. Implementation. Explain the main computational choices.
  5. Parameter study. Show how reconstruction changes as a parameter changes.
  6. Results. Present quantitative and visual comparisons.
  7. Robustness and failure. Show stress tests and at least one failure case.
  8. Limitations. Explain when the method should not be trusted.
  9. LLM-use statement. Disclose how the required LLM support was used.

Claim-Evidence Table

Each project should turn its results into claims that are supported by evidence.

Claim Evidence Limitation or Failure
Example: TV preserves the main edge better than quadratic smoothing. Gradient maps and edge contrast across three noise levels. TV creates staircasing on smooth ramps.
Example: The sparse DCT method works with 40% observed pixels. RMSE and visual comparison across five masks. It fails on texture-rich images with non-sparse DCT coefficients.
Example: A learned prior improves denoising on similar images. Test-set RMSE on held-out crops from the same image family. It degrades under domain shift to a different image family.

The table protects against overclaiming. A reconstruction can look impressive while supporting only a modest claim.

Possible Project Topics

Teams may choose one of the following topics or propose a related topic.

Topic Possible Methods Reliability Question
Image deblurring Tikhonov vs TV What happens when the blur kernel is wrong?
Image inpainting interpolation vs sparse DCT or TV How many missing pixels can be tolerated?
Image denoising Gaussian smoothing vs wavelets vs TV Which details are removed with the noise?
Poisson image denoising Gaussian model vs Poisson likelihood When does the Gaussian approximation fail?
Sparse reconstruction minimum norm vs l1 regularization How many measurements are needed?
Wavelet denoising hard vs soft thresholding How should the threshold be chosen?
Plug-and-play reconstruction classical regularization vs denoiser prior Is the output supported by the data?
Small-feature preservation smoothing vs TV vs wavelets Does the method erase weak structures?
Domain shift in denoising train/test on different image families Does a learned prior generalize?
Robustness audit compare several methods under perturbations Which method fails most gracefully?

The instructor may provide additional test images or degraded observations. These may be used to test whether the submitted notebook generalizes beyond the examples used during development.

Choosing a Good Topic

A good topic is small enough to test carefully.

Prefer:

  • one clear degradation;
  • one or two image families;
  • two methods that make different assumptions;
  • a metric connected to the scientific question;
  • a failure case that can be reproduced.

Avoid:

  • comparing many methods superficially;
  • using a neural network without explaining its training data or prior;
  • reporting only visual quality;
  • using a black-box result that no teammate can explain;
  • presenting an LLM-generated notebook that has not been independently tested.

Grading Rubric

The syllabus assigns 30% of the course grade to the two-student team modeling project, including the joint report and individual oral defense. The percentages below describe the internal breakdown of that 30% project grade; they do not add any graded work outside the syllabus.

Component Weight
Mathematical model and assumptions 15%
Reproducible notebook and implementation 15%
Experiments and parameter study 15%
Robustness and failure analysis 20%
Written report clarity 10%
Oral defense and code walkthrough 25%
LLM-use statement Required, ungraded

Oral Defense

The oral defense is individual. Each student must be able to explain the whole project, not only the part they personally wrote.

During the defense, students may be asked to:

  • define the forward model;
  • explain why the inverse problem is difficult;
  • derive or interpret the reconstruction objective;
  • explain how a parameter affects the result;
  • interpret a quantitative metric;
  • explain a figure from the report;
  • identify one failure case;
  • walk through selected lines of code;
  • run the notebook on a new test image or modified degradation;
  • explain how LLM assistance was checked.

The oral defense is worth 25% of the project grade because it tests ownership of the submitted artifact. It is the part of the project that cannot be outsourced to an LLM.

Defense Preparation Checklist

Each student should be ready to answer:

  • What does x represent in your project?
  • What does y represent?
  • What is known, unknown, or estimated in the forward model?
  • Which method is your baseline?
  • Which method uses a stronger prior?
  • Which parameter matters most?
  • What happens when that parameter is too small or too large?
  • Which result is supported by the data?
  • Which result is mostly imposed by the prior?
  • What is your clearest failure case?
  • Which part of the code would you change first if the method failed on a new image?
  • What did the LLM help with, and how did you verify it?

What A Strong Project Shows

A strong project does not simply show that one method looks good on one image.

It shows:

  • a clear mathematical model;
  • correct and reproducible computation;
  • thoughtful parameter choices;
  • comparison against a meaningful baseline;
  • evidence from more than one test condition;
  • an honest failure case;
  • clear explanation of what the method can and cannot claim.

In mathematical imaging, a reconstruction is not only an image. It is an argument supported by a model, an algorithm, experiments, and interpretation.
